Chào mừng thầy và các bạn đến với buổi thuyết trình về đồ án Hệ thống Quản lý Khách sạn. Chúng em tập trung vào việc áp dụng các nguyên lý thiết kế phần mềm hiện đại và 4 tính chất cốt lõi của OOP.

Giới thiệu Bối cảnh & Bài toán

Bối cảnh

Ngành khách sạn cần quản lý chính xác phòng, đặt phòng và khách hàng. Hệ thống cũ thủ công gây sai sót.

Bài toán

Xây dựng phần mềm quản lý hỗ trợ đầy đủ CRUD. Áp dụng triệt để 4 tính chất OOP.

Bài toán đặt ra không chỉ là một phần mềm chạy được, mà là một hệ thống có kiến trúc tốt, dễ dàng mở rộng và bảo trì nhờ vào việc áp dụng các chuẩn mực của lập trình hướng đối tượng.

Đặc tả Actor (Tác nhân)

Quản lý

Quản lý phòng, báo cáo doanh thu, nhân sự.

Lễ tân

Đặt phòng, Check-in/out, Quản lý khách.

Dịch vụ

Cung cấp dịch vụ cho khách.
Việc chia tách Actor giúp chúng em xác định rõ phạm vi trách nhiệm và phân quyền trong hệ thống. Lễ tân tập trung vào quy trình nghiệp vụ, trong khi Quản lý tập trung vào dữ liệu chiến lược.

Đây là bức tranh tổng thể. Mọi tương tác của người dùng đều được đóng gói trong các Use Case này để đảm bảo đạt được mục tiêu cuối cùng của từng Actor.

Đặc tả UC Quan trọng: "Đăng nhập"

Tiêu chí Nội dung
Tác nhân Tất cả người dùng (Admin, Staff, Service)
Mô tả Xác thực danh tính và cấp quyền truy cập.
Luồng chính Nhập User/Pass -> Kiểm tra -> Chuyển đến Dashboard tương ứng.

Tại sao quan trọng? Đảm bảo tính bảo mật (Encapsulation) và cá nhân hóa giao diện.

Use Case Đăng nhập là cửa ngõ của hệ thống. Tại đây, chúng em áp dụng phân quyền để đảm bảo nhân viên lễ tân không thể xóa phòng.

Đặc tả UC Phức tạp: "Đặt phòng"

Kiểm tra phòng trống (Availability): Logic phức tạp để tránh trùng lặp ngày.

Tính giá: Dựa trên loại phòng (Standard/VIP) và số đêm.

Tạo Booking: Liên kết Khách hàng và Phòng.


// Logic kiểm tra phòng trống
boolean isAvailable = bookings.stream().noneMatch(b -> 
    !(checkOut.isBefore(b.getCheckIn()) || checkIn.isAfter(b.getCheckOut()))
);
Đây là nghiệp vụ trọng tâm. Hệ thống phải đảm bảo không xảy ra tình trạng "Double-booking" và giá tiền phải được tính toán chính xác.

Biểu đồ này thể hiện cách các đối tượng giao tiếp. AuthService đóng vai trò trung gian để tách biệt giao diện và dữ liệu.

Sequence Diagram: Đặt phòng

Phân tích & Chia lớp đối tượng

  • Room: Quản lý trạng thái, giá.
  • Customer: Thông tin cá nhân.
  • Booking: Liên kết Room & Customer.
  • Invoice: Tính toán tài chính.
Nguyên lý Single Responsibility (SRP):
Mỗi class chỉ chịu trách nhiệm cho một thực thể duy nhất trong thế giới thực.
Việc chia class dựa trên các thực thể thế giới thực. Mỗi đối tượng chỉ chịu trách nhiệm cho một công việc duy nhất.

Kế thừa & Đa hình (Room)

classDiagram class Room { <> +calculatePrice()* } class StandardRoom { +calculatePrice() (x1.0) } class VIPRoom { +calculatePrice() (x1.2) } Room <|-- StandardRoom Room <|-- VIPRoom

Sử dụng Abstract Class để định nghĩa khung sườn chung. Tính đa hình giúp tính giá linh hoạt.

Nhờ kế thừa, chúng em tránh lặp code. Tính đa hình cho phép hệ thống tính giá chính xác mà không cần dùng nhiều lệnh IF-ELSE.

Quan hệ Entity (Composition)

classDiagram class Booking { -bookingId } class Customer { -name } class Room { -roomId } Booking *-- Customer : contains Booking *-- Room : contains

Tại sao dùng Composition? Vì một Booking "có" một Khách hàng và "có" một Phòng, không phải "là" chúng.

Chúng em ưu tiên Composition thay vì Inheritance cho mối quan hệ này để tạo sự linh hoạt.

Interface & Design Pattern

Interfaces

  • IManageable<T>: Chuẩn hóa CRUD.
  • ISearchable<T>: Tìm kiếm đa tiêu chí.
  • IStorable: Lưu trữ dữ liệu.

Patterns

  • Singleton: RoomManager (1 instance).
  • Factory: RoomFactory (Tạo object).
  • MVC: Tách biệt UI và Logic.
Interface giúp định nghĩa hợp đồng cho hệ thống. Pattern giúp code sạch và dễ bảo trì hơn.

Kiến trúc MVC (4 Tầng)

PRESENTATION (View): Swing UI
⬇
BUSINESS (Controller): Managers
⬇
MODEL: Entities (Room, Booking)
⬇
DATA (Storage): JSON Files
Mô hình MVC kinh điển. Nếu muốn đổi giao diện sang Web, chỉ cần viết lại tầng Presentation.
